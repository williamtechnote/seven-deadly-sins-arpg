# 事件房补给与祈愿扩展 Design

## 背景

当前事件房系统已经具备单房抽取、双选项结算、HUD 摘要、存档持久化与基础回归检查，但 `TODO.md` 的 `Active` 为空，说明下一轮 heartbeat 需要先重新排队再继续扩展。事件房方向仍是当前最连续的 Phase 2 主题，因此本轮优先补充两个“低接线成本、可直接复用现有框架”的原型。

## 目标

- 新增 `战备商柜`：提供花金币换立即可用消耗品的双选项。
- 新增 `祈愿圣坛`：提供两种不同的本局增益路线，复用现有 `runEffects`。
- 保持事件房效果尽量收敛到 `shared/game-core.js`，让 `game.js` 只负责展示与应用结果。
- 将第三个后续项保留在 `TODO.md`：补充 HUD 的事件房类型标签与路线摘要。

## 方案选择

### 方案 A：继续只做数值不同、效果类型不变的房间

- 优点：改动最少。
- 缺点：新原型会和现有 `赌徒/泉眼/血契` 过于相似，扩展价值偏低。

### 方案 B：补一个“即时资源交易”房间 + 一个“运行期增益”房间（推荐）

- `战备商柜` 需要一个新的共享效果类型，用金币换固定道具。
- `祈愿圣坛` 继续复用 `runEffectBuff`，但增益方向改成体力恢复 / 特攻节律，而不是单纯增伤。
- 优点：一项扩展即时收益结算，一项扩展 run effect 覆盖面，测试面清晰且不需要大规模重构。

### 方案 C：直接把 HUD 路线摘要也一起做完

- 优点：体验更完整。
- 缺点：本轮 heartbeat 目标要求实现最早两个 TODO；HUD 摘要可以保留为下一轮，不必把 UI 改动和新房型耦合在一起。

## 本轮设计

- 采用方案 B。
- `战备商柜`
  - 选项 1：花 45 金币换 1 瓶 `净化药剂`。
  - 选项 2：花 60 金币换 1 瓶 `狂战油`。
  - 共享逻辑新增“金币换道具”结算，并在结果文案里同步体现消耗与获得。
- `祈愿圣坛`
  - 选项 1：`复苏祷言`，本局体力恢复提高。
  - 选项 2：`迅击祷言`，本局特攻冷却缩短。
  - 继续使用 `runEffectBuff`，以便自动参与 `GameState.refreshRunEffects()`。
- 事件房样式继续沿用现有 `type` 映射，不为本轮额外增加复杂 UI 状态；HUD 摘要项留在下一轮。

## 测试

- 回归检查先新增并验证失败：
  - `战备商柜` 是否暴露双选项、扣除正确金币、发放正确道具。
  - 金币不足时是否拒绝结算，并返回明确原因。
  - `祈愿圣坛` 是否暴露双选项、持久化选项并正确生成对应 `runEffects`。
- 完成后运行固定 heartbeat 校验命令，作为提交与合并依据。
