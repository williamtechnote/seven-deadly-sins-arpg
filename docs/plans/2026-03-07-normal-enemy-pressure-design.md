# 普通小怪压迫修正与战斗回归设计

## 背景

当前 Active TODO 的前两项都围绕同一条战斗基线：

- 普通小怪压迫略高，需要下调基础移动速度约 6%。
- 调整后需要补一条可重复的回归校验，确认前期长剑更容易打出无伤先手，同时 Boss 难度没有被一并削弱。

## 目标

- 仅下调普通小怪的基础移动速度。
- Boss 速度与行为保持不变。
- 用 CLI 回归脚本锁定这条平衡意图，避免后续继续漂移。

## 方案对比

### 方案 A：直接逐个下调 `data.js` 中所有普通小怪速度（推荐）

- 在 `ENEMIES` 配置层统一按约 6% 下调并取整。
- Boss 数据完全不动。
- 回归脚本直接读取 `data.js` 并做轻量战斗模型验证。

优点：
- 改动最小，风险集中在数据层。
- 游戏运行时逻辑无需新增条件分支。

缺点：
- 之后若再做全局难度系数，需要继续依赖脚本兜底。

### 方案 B：在 `Enemy` 构造时额外乘一个常量

- 运行时给普通小怪统一乘上一个 `0.94` 系数。
- Boss 构造不接该系数。

优点：
- 后续全局调参更集中。

缺点：
- 会把“基线速度”藏进运行时代码，和 `data.js` 的展示值脱节。

### 方案 C：只提高长剑安全距离，不碰小怪速度

优点：
- 几乎零风险。

缺点：
- 无法满足当前 TODO 的“小怪压迫修正”目标。

## 结论

采用 **方案 A**：

- `data.js` 直接下调普通小怪 `speed`。
- `scripts/regression-checks.mjs` 新增两类保护：
  - 普通小怪速度基线与 Boss 速度不变。
  - 一个 deterministic 的开场接敌模型，验证长剑前期无伤先手窗口相对旧基线提升，且 Boss 先手窗口没有同步扩大。

## 验收标准

- 所有 `ENEMIES[*].speed` 相比上一基线约下降 6%，Boss 速度保持原值。
- 回归脚本能证明普通小怪侧的安全先手窗口变大。
- 同一脚本能证明 Boss 侧窗口变化极小或为零。
- README / TODO / PROGRESS.log 同步更新。
