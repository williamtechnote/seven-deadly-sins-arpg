# 战斗反馈：状态 HUD 与消耗品反馈 Design

## 背景

当前战斗反馈已经覆盖命中冲击环、伤害数字、低血量/低体力警示，但还有两个明显缺口：

1. 快捷栏药剂在资源已满时仍会被消耗，且玩家看不到实际恢复了多少。
2. HUD 状态信息集中在一行展示，负面效果、增益和抗性混在一起，读起来不够快。

## 目标

- 治疗/体力药剂只在产生实际收益时消耗，并给出明确的恢复反馈。
- HUD 将状态拆分为“负面状态”和“增益状态”两组，优先显示更紧急、剩余时间更短的条目。
- 保持改动最小，尽量把可验证逻辑沉到 `shared/game-core.js`，继续复用现有 `scripts/regression-checks.mjs`。

## 方案选择

### 方案 A：纯前端散落实现

- 直接在 `game.js` 中加条件判断和文本拼接。
- 优点：改动路径短。
- 缺点：难测试，后续 HUD 文案和道具行为会继续分散。

### 方案 B：共享纯逻辑 helper + 场景薄接线（推荐）

- 在 `shared/game-core.js` 提供：
  - 消耗品结算 helper，返回是否消耗、实际恢复量和反馈文案类型。
  - 状态 HUD 汇总 helper，负责分组、排序和格式化。
- `game.js` 只负责调用 helper 和把结果渲染到浮字/HUD。
- 优点：可通过现有回归脚本验证，保持运行时逻辑简单。
- 缺点：需要多改一个共享文件。

### 方案 C：完整 UI 重做

- 引入图标化状态条、动画残影、专门提示层。
- 优点：视觉提升更明显。
- 缺点：超出当前最小迭代范围，不适合这一轮。

## 本轮设计

- 采用方案 B。
- 消耗品反馈：
  - `healHp` / `healStamina` 计算实际恢复量。
  - 若恢复量为 0，则不消耗并返回“生命已满”或“体力已满”。
  - 若成功恢复，则返回数值化反馈，供场景显示 `+HP` / `+ST` 浮字。
- 状态 HUD：
  - 负面：`burn / bleed / slow / reverseControl`
  - 增益：`statusResistance / damageBuff`
  - 同组内按剩余时间升序排序，优先突出更急迫的状态。
  - HUD 拆为两行文本，分别使用危险色与增益色。

## 测试

- 为共享 helper 新增回归测试：
  - 满血/满体力不消耗药剂。
  - 部分恢复时返回正确数值与下一状态。
  - 状态 HUD 分组与排序稳定。
