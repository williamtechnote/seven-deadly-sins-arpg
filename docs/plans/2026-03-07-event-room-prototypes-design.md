# 事件房原型扩展 Design

## 背景

`赌徒圣坛` 已经把事件房框架、结算面板和存档持久化打通，但 `TODO.md` 里仍有两个未完成的原型：`疗愈泉眼` 和 `血契祭坛`。本轮 heartbeat 需要直接沿用现有框架，把剩余原型做成可交互、可持久化、可回归验证的完整闭环。

## 目标

- 为 `疗愈泉眼` 提供双选项治疗型结算，并支持恢复/净化两类即时收益。
- 为 `血契祭坛` 提供双选项风险增伤结算，并将本局增伤/承伤变化纳入 `runEffects`。
- 保持事件房逻辑尽量沉到 `shared/game-core.js`，避免在 `game.js` 中硬编码每个房间的数值规则。

## 方案选择

### 方案 A：场景里按房间 key 写死逻辑

- 优点：接线最快。
- 缺点：结算规则与运行时增益无法复用，也难通过 CLI 回归检查验证。

### 方案 B：扩展共享事件房定义与效果汇总 helper（推荐）

- 在 `RUN_EVENT_ROOM_POOL` 中为两个新房间补齐双选项定义。
- 在 `shared/game-core.js` 中统一结算即时收益，并新增“已结算事件房对本局倍率的影响” helper。
- `game.js` 只负责展示、调用结算结果、刷新玩家状态和 HUD。
- 优点：与现有 run modifier 架构一致，易测试、易持久化。

### 方案 C：把事件房与 run modifier 系统完全合并

- 优点：长期上更统一。
- 缺点：本轮 heartbeat 不需要重构到这个粒度，收益不足。

## 本轮设计

- 采用方案 B。
- `疗愈泉眼`
  - 选项 1：大量恢复当前生命。
  - 选项 2：少量恢复并清除当前负面状态。
- `血契祭坛`
  - 选项 1：高风险高收益，增伤更高且承伤提升更多。
  - 选项 2：稳健契约，小幅增伤并附带较轻承伤惩罚。
- `runEffects` 由 `run modifier` 和 `event room` 共同决定，因此在事件房结算后需要立刻刷新。
- LevelScene 的事件房外观与浮字反馈改为读取当前事件房定义，而不是只识别 `赌徒圣坛`。

## 测试

- 回归检查新增：
  - `疗愈泉眼` 选项是否按预期恢复生命，并在需要时清除负面状态。
  - `血契祭坛` 选项是否持久化选择结果，并正确影响本局伤害/承伤倍率。
  - 已结算事件房在反序列化后仍能恢复对应名称、描述与效果。
